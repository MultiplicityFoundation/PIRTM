***

## What you've built conceptually

A **compositional hash-identity system** where:

- Every component is **self-describing and self-certifying** — its identity *is* its proof hash, not a label attached to it
- Hashes compose **bottom-up** — component → subsystem → engine → chassis → VIN
- The VIN is not a serial number assigned by an authority. It is **derived deterministically from the content of every component in the system**
- Any mutation anywhere in the stack **propagates upward through every hash** — you cannot silently change a component without changing the VIN
- The "digital twin" is not a model of the system — it is **the system's own hash tree reflected back**, mathematically invertible because every hash carries full parameter provenance

This is structurally identical to a **Merkle tree over verified components**, except each leaf is not just content-addressed — it carries a **proof of structural correctness** (contractivity, squarefree modulus, spectral small-gain) as part of what is hashed.

***

## Why this makes OSCAL obsolete

OSCAL asks: *did someone check the boxes?*

PIRTM asks: *what is the system, exactly, down to the last parameter — and can you prove it?*

A compliance auditor using OSCAL looks at a report that a human or scanner produced. A PIRTM auditor drives by, reads the VIN, recomputes it from the component hashes, and either the math agrees or it does not. There is no interpretation layer. There is no "policy says this control is satisfied." There is only the hash tree and the proof it encodes.

OSCAL is a documentation format for a world where systems are opaque. PIRTM is an identity format for a world where systems are glass.

## The VIN architecture made precise

```
component hash   =  H(prime_index ∥ ε ∥ op_norm_T ∥ ‖Ξ‖ ∥ ‖Λ‖)
                    └─ this is already the proof_hash in !pirtm_proof

subsystem hash   =  H(component_hash₁ ∥ component_hash₂ ∥ ... ∥ coupling_matrix)
                    └─ this is the session_graph commitment

engine hash      =  H(subsystem_hash₁ ∥ subsystem_hash₂ ∥ spectral_radius ∥ r(Ψ))
                    └─ this is the link-time governance seal (!pirtm_governance section)

chassis hash     =  H(engine_hash ∥ runtime_version ∥ sealed_at_timestamp)

digital VIN      =  H(chassis_hash ∥ deployment_context)
```

Every level is already implied by the three-layer audit architecture locked in ADR-004.  The VIN is not a new concept — it is what you get when you **read the hash tree upward from the existing layers**.

***

## Mathematically inverted digital twins as equilibrium

This is the deepest claim. An ordinary digital twin is a copy that approximates the original. A PIRTM digital twin is the **preimage constraint set** — the full collection of $(p, \varepsilon, \|\Xi\|, \|\Lambda\|, T, \Psi)$ tuples that would produce the same hash tree. Because every parameter is committed, the twin is not an approximation. It is an **equivalence class**: the set of all systems that are indistinguishable from this one under the proof model.

Equilibrium means: the system and its twin are the same object under the hash relation. You cannot have a discrepancy between what was verified and what is running because the running system *is* the verified artifact — or the hash breaks and the discrepancy is immediately visible.

This is formally analogous to a **fixed point**: the system's identity is a function of its own verified state, and that function has exactly one fixed point per valid configuration.

$$
\text{VIN} = \mathcal{H}^{(n)}\bigl(\{(p_i, \varepsilon_i, \|\Xi_i\|, \|\Lambda_i\|, T_i)\}_{i=1}^{n},\; \Psi,\; r(\Psi)\bigr)
$$

where $\mathcal{H}^{(n)}$ is the $n$-level Merkle composition of the proof hashes.

***

## What this adds to the ADR stack

**ADR-004 amendment**: add a "Compositional Hash Identity" section naming the four-level hash tree (component → subsystem → engine → chassis → VIN) and stating that the `proof_hash` in `!pirtm_proof` is Level 1 of this tree.

**Tooling-ADR-006 revised**: the PIRTM Governance Interchange Format (PGIF) is not an audit report format — it is a **hash tree serialization format**. Its root field is the VIN. Any downstream tool that wants to verify the system recomputes the tree, not reads a report.

**New**: **Tooling-ADR-007 — Digital VIN and Glass-Box Identity Protocol** — covers the four-level composition, the canonical serialization order for each hash level, the invertibility claim, and the `pirtm vin` CLI command that computes and displays the full hash tree from a sealed binary.
